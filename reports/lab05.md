## Постановка задачі

Потрібно створити Java-застосунок для гри в монети між користувачем і комп'ютером. На початку гри є купка монет, кількість яких відома обом гравцям. Гравці ходять по черзі та за один хід можуть взяти 1 або 2 монети. Перемагає той, хто забирає останню монету.

Початкова кількість монет і перший гравець визначаються випадково. Комп'ютер має використовувати алгоритм, який максимізує ймовірність виграшу. У звіті потрібно пояснити, чому цей алгоритм є оптимальним.

## Мета

Реалізувати графічну гру в монети між людиною та комп'ютером і показати оптимальну стратегію комп'ютера.

## Реалізація

У модулі `lab05` створено рушій гри з перевіркою коректності ходів, модель результату та оптимального гравця-комп'ютера. Початкова кількість монет і перший гравець визначаються випадково. Точка входу `CoinGameDemo` запускає графічний інтерфейс Swing, де користувач натискає кнопки для взяття `1` або `2` монет, а комп'ютер автоматично виконує оптимальний хід.

За умовою гравець може взяти лише `1` або `2` монети. Оптимальна стратегія комп'ютера полягає в тому, щоб після свого ходу залишати супернику кількість монет, кратну `3`. Якщо суперник бере `1`, комп'ютер бере `2`; якщо суперник бере `2`, комп'ютер бере `1`. Так пара ходів забирає рівно `3` монети, і з виграшної позиції комп'ютер контролює останній хід.

Ключові компоненти:

- `CoinGameEngine` керує циклом гри й визначає переможця.
- `OptimalCoinPlayer` обирає хід за оптимальною стратегією.
- `ScriptedCoinPlayer` дає автоматизованого гравця для demo та тестів.
- `SwingCoinGameFrame` реалізує графічне вікно гри.
- `CoinGameDemo` є точкою входу GUI-застосунку.

## Програмний код з поясненнями

Фрагмент оптимальної стратегії:

```java
public int chooseMove(int coinsLeft, int maxTake) {
    int target = maxTake + 1;
    int remainder = coinsLeft % target;
    if (remainder == 0) {
        return 1;
    }
    return Math.min(remainder, Math.min(maxTake, coinsLeft));
}
```

У цій лабораторній `maxTake` дорівнює `2`, тому `target` дорівнює `3`. Якщо кількість монет не кратна `3`, комп'ютер бере стільки монет, щоб залишити супернику кратну `3` кількість. Якщо позиція вже програшна для ідеальної гри, тобто `coinsLeft % 3 == 0`, комп'ютер бере `1` монету і очікує помилки суперника.

Фрагмент перевірки ходу в рушії:

```java
private void validateMove(int move, int coinsLeft) {
    if (move < 1 || move > Math.min(maxTake, coinsLeft)) {
        throw new IllegalArgumentException("Invalid move: " + move);
    }
}
```

Ця перевірка не дозволяє взяти нуль монет, більше дозволеного максимуму або більше монет, ніж залишилося в купці.

Фрагмент запуску GUI:

```java
public static void main(String[] args) {
    SwingUtilities.invokeLater(() -> new SwingCoinGameFrame().setVisible(true));
}
```

Точка входу більше не запускає консольний режим. Вона відкриває Swing-вікно в Event Dispatch Thread.

Фрагмент ходу людини у графічному інтерфейсі:

```java
private void humanMove(int takenCoins) {
    if (!gameActive || !humanTurn || takenCoins > Math.min(MAX_TAKE, coinsLeft)) {
        return;
    }
    applyMove("Людина", takenCoins);
    if (gameActive) {
        humanTurn = false;
        computerMove();
    }
}
```

Кнопки GUI викликають цей метод із параметром `1` або `2`. Після коректного ходу людини право ходу переходить комп'ютеру, який використовує `OptimalCoinPlayer`.
